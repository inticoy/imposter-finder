# Interactive Bot Design (@멘션 전적 조회)

Discord에서 봇을 `@멘션`하면 자연어/명령어로 **유저별 최근 전적·평균딜·실력 등급(S/A/B/C/F)**을
응답하는 인터랙티브 기능의 설계 문서. 아직 구현 전이며, 방향과 명세만 확정한 상태다.

이 문서는 [implementation-plan.md](implementation-plan.md)의 자동 알림(범인찾기)과 **별개로 동작하는**
대화형(query) 기능을 다룬다.

## 배경 / 현재 구조의 제약

현재 봇은 GitHub Actions cron으로 주기 실행되는 **배치 방식**이다.

- `imposter_finder/discord.py`는 REST API로 `POST /messages`만 보낸다 (메시지 전송 전용).
- 실행 후 프로세스가 종료되므로, Discord에서 누가 멘션해도 그것을 **듣고 있는 프로세스가 없다.**

따라서 `@멘션` 실시간 응답을 위해서는 배치(Actions)만으로는 불가능하고, 별도의 수신 경로가 필요하다.

### 수신 방식 비교

| 방식 | 동작 | 실시간 멘션 | 자연어 | 호스팅 | Actions로 가능? |
|---|---|---|---|---|---|
| **A. 상주형 (Gateway/WebSocket)** | 봇이 Discord Gateway에 항상 연결, `MESSAGE_CREATE` 수신 | ✅ `@멘션` 그대로 | ✅ 원문 전체 | 24시간 상주 프로세스 필요 | ❌ |
| **B. 서버리스 Slash Command** | `/전적 홍길동` 형태, Discord가 HTTP 엔드포인트로 POST | ⚠️ 멘션 아님 | ⚠️ 인자만 | 서버리스(무료 가능) | 간접 가능 |

**결정: A(상주형).** `@살래말래 홍길동 어때?` 같은 멘션 + 자연어를 그대로 살리려면 Gateway 수신이 필요하다.
Discord 개발자 포털에서 **Message Content Intent**를 켜야 한다.

## 전체 아키텍처

```
players.json ─┐
              ├─► [기존] Actions cron → 범인찾기 자동 알림   (그대로 유지)
PubgClient ───┤
registry ─────┤
analysis  ────┘        (공유 모듈로 재사용)
              └─► [신규] 상주형 봇 (discord.py)
                    Gateway 연결 → @멘션 수신 → 의도 파싱
                    → 닉네임 해석 → 최근 N판 집계 → 등급 → embed 응답
                    (+ 캐시로 rate limit 방어)
```

기존 배치 코드는 수정하지 않고, `PubgClient`·`registry`·점수 로직만 **공유**해서 재사용한다.

## 신규 컴포넌트

### 1) 상주 프로세스 (`bot.py`)

- `discord.py` 라이브러리 사용. (기존 raw `urllib` 대신. 상주 봇은 Gateway 연결·이벤트·재연결 처리가
  필요해 라이브러리가 유리하다. 전송은 기존 `DiscordClient`를 그대로 써도 된다.)
- `on_message`에서 봇 멘션 여부 확인 → 텍스트 파싱.
- Discord 개발자 포털에서 **Message Content Intent ON** (필수).
- `DISCORD_BOT_TOKEN`은 기존 것 재사용 가능.

### 2) 의도 파싱 (초기: 규칙 기반)

| 입력 예시 | 해석 |
|---|---|
| `@살래말래 내 전적` | 멘션한 사람의 `discord_user_id`로 닉 조회 |
| `@살래말래 @홍길동 어때` | 멘션 대상 `user_id`로 닉 조회 |
| `@살래말래 아무개 평균딜` | 텍스트에서 닉네임 추출 |

- 키워드: `전적 / 평균딜 / 등급 / 실력 / 최근`. 없으면 기본 = 종합 카드.
- `registry.py`에 `discord_user_id → PubgPlayer` 역방향 조회를 추가한다 (필드는 이미 존재).
- LLM 기반 의도 파싱은 이후 개선 항목. 초기엔 규칙으로 충분.

### 3) 최근 N판 집계 + 등급 산정 (`analysis`에 함수 추가)

- 데이터 기준: **최근 N판 집계** (딜·킬·등수).
- 흐름: `PubgClient.lookup_players()`로 최근 매치 ID 목록 확보 → 앞 N개(예: 10)를 `get_match()`로 조회
  → 각 매치에서 본인 participant stats 추출 → 집계.
- 집계 지표: **평균딜 / 평균킬 / 평균등수 / KD / 승률**, 그리고 최근 폼(최근 3판 평균딜 등).
- 등급 매핑 (예시 값, 실제 임계값은 친구들 데이터로 조정):

  | 등급 | 대략 기준 (예시) |
  |---|---|
  | **S** | 평균딜 ≥ 350, KD ≥ 3 |
  | **A** | 평균딜 ≥ 250 |
  | **B** | 평균딜 ≥ 150 |
  | **C** | 평균딜 ≥ 80 |
  | **F** | 그 미만 |

- 기존 `analysis/pubg.py`의 `PlayerScore`·`adjust()`·`stars()` 톤을 재사용해 등급별 한줄평도 붙일 수 있다.

### 4) 캐시 (rate limit 방어 — 필수)

- PUBG API는 **분당 10요청** 제한. 최근 10판 집계 = 한 명당 약 11요청 → 한 번 조회로 한도 소진.
- 인메모리 dict `{nickname: (timestamp, 집계결과)}`, TTL 5~10분. 상주 프로세스이므로 인메모리로 충분.
- 매치 상세도 `{match_id: match}`로 캐시하면 친구들끼리 같은 판이 겹칠 때 재활용된다.

## 예상 응답 (embed)

```
🎯 홍길동 (steam) · 최근 10판
등급: A  ⭐⭐⭐⭐🫥
평균딜 268 · 평균 2.1킬 · 평균 등수 #14 · KD 2.4 · 승률 20%
최근 폼: 상승세 📈  (최근 3판 평균딜 320)
```

## 호스팅 (무료 24시간 상주)

상주 봇은 Gateway를 계속 유지해야 하므로, 유휴 시 잠드는(sleep) 무료 티어는 부적합하다.
"항상 켜져 있는 무료" 옵션만 후보로 둔다.

| 옵션 | 무료 성격 | 상주 적합 | 난이도 | 비고 |
|---|---|---|---|---|
| **Oracle Cloud Always Free** (권장) | 평생 무료 | ✅ 완전 상주 | 중 (VM 직접 관리) | ARM Ampere 또는 AMD micro. 카드 등록 필요. 이 용도엔 스펙 충분 |
| **집 하드웨어** (라즈베리파이/미니PC/방치 노트북) | 전기값만 | ✅ 완전 상주 | 하 | 있으면 최우선. `systemd`로 등록 |
| **Google Cloud e2-micro Always Free** | 평생 무료 | ✅ 상주 | 중 | 특정 US 리전 1대, 1 vCPU/1GB로 충분. 카드 필요 |
| Fly.io | 사용량 크레딧 | ⚠️ 가능하나 순수 무료 아님 | 중 | 소규모는 사실상 무료권, 한도 관리 필요 |
| Railway / Render / Replit | ❌ 부적합 | ❌ | 하 | 무료는 sleep 또는 상시 유료. Discord 상주엔 비추 |

**결정 가이드:** 집에 남는 하드웨어가 있으면 그것이 1순위(제일 간단·안정). 새로 구한다면
**Oracle Cloud Always Free**가 무료 중 가장 확실하다.

> 보안: 봇 토큰·`players.json`은 환경변수/시크릿으로만 두고 리포에 커밋하지 않는다 (현재 방침 유지).

## 트레이드오프 요약

- **장점:** 멘션·자연어를 그대로 살림. 기존 자산(`PubgClient`·`registry`·점수 로직) 대거 재사용.
  `discord_user_id` 매핑이 이미 있어 "내 전적"이 바로 된다.
- **비용:** 24시간 상주 프로세스 1개(무료 티어로 커버 가능) + 캐싱 관리 필요.

## 열린 항목 / 이후 개선

- 등급 임계값을 친구들 실제 데이터로 캘리브레이션.
- 랭크/일반 구분이 필요하면 시즌 누적 스탯(`/players/{id}/seasons/{seasonId}`) 병행 검토.
- 규칙 기반 → LLM 의도 파싱 확장.
- 조회 결과/등급 이력을 SQLite에 저장할지 (implementation-plan.md의 스키마와 통합 여부).
